## WORKFLOW GAP ANALYSIS

### CURRENT WORKFLOW:
1. Client posts task � status: "open"
2. Tasker **DIRECTLY ACCEPTS** task � status: "in_progress" (tasker_username assigned)
3. Client marks complete � status: "completed"

### DESIRED WORKFLOW (PER USER REQUIREMENTS):
1. Client posts task � status: "open"
2. Tasker **OFFERS TO TAKE** task � status: needs intermediate state or offer tracking
3. **CLIENT REVIEWS AND ACCEPTS** the tasker's offer � status: "booked" or "in_progress"
4. **EITHER client OR tasker** marks complete � status: "completed"

### KEY DIFFERENCES:
1. L Missing: Two-step acceptance process (offer � client approval)
2. L Missing: Tasker ability to mark tasks complete

## USER MODEL ARCHITECTURE (TASKER-SIDED APPROACH)

### CURRENT MODEL:
- Users choose `role: "client"` OR `role: "tasker"` at registration
- Roles are mutually exclusive
- Role determines all permissions

### DESIRED MODEL (PER MENTOR GUIDANCE):
- **All users are clients by default** - can post tasks, hire taskers
- **Users can optionally become taskers** - sign up separately to accept tasks
- This mirrors platforms like Uber (everyone can ride, drivers sign up separately)

### IMPLEMENTATION REQUIREMENTS:
1. **User Schema Changes:**
   - Add `is_tasker: bool = False` to user document
   - Add `phone: str` field
   - Add `zip_code: str` field (replaces general location for clients)
   - Keep tasker-specific fields (skills, service_categories, etc.) populated when becoming tasker

2. **Client Registration Fields (Required for all users):**
   - `full_name` - User's name
   - `username` - Email address (used for login)
   - `phone` - Phone number
   - `password` - Account password
   - `zip_code` - User's zip code

3. **Tasker Enrollment Fields (TBD):**
   - Additional information required when becoming a tasker
   - Likely: skills, service_categories, bio, profile_image_url
   - To be finalized based on what information is needed to match taskers with jobs

5. **New "Become a Tasker" Flow:**
   - Endpoint: `POST /users/me/become-tasker`
   - Collects: skills, service_categories, etc. (TBD)
   - Sets `is_tasker: True`

6. **Authorization Logic Changes:**
   - Client actions (create task, search taskers): Any authenticated user
   - Tasker actions (view open tasks, accept tasks): Only if `is_tasker == True`

7. **Database Migration:**
   - Existing `role: "tasker"` users → `is_tasker: True`
   - Existing `role: "client"` users → `is_tasker: False`

### IMPLEMENTATION TIMING:
This should be implemented before or alongside Phase 3 (Tasker Profile Enhancement), as it fundamentally changes how tasker profiles work.

     ```json
       "suggested_category_id": "...",
       "suggested_task_type": "General furniture assembly",
       "suggested_task_type_id": "...",
       "all_matches": [...] // Array of matching task types
     }
     ```
   - Handle GOOGLE_API_KEY missing gracefully

3. **Add Pydantic Models:**
   ```python
   class AISearchQuery(BaseModel):

       query: str

       suggested_category_id: Optional[str]
       suggested_task_type: Optional[str]
       suggested_task_type_id: Optional[str]
   ```

## END OF CATEGORIZATION & SEARCH IMPLEMENTATION PLAN

config.py

  Purpose: Loads all settings and external service configurations.

  Contains:
  - Environment variables (SECRET_KEY, DATABASE_URL, GOOGLE_API_KEY)
  - JWT settings (algorithm, expiration time)
  - Password hasher setup
  - Gemini AI model initialization

  Why separate: Settings are loaded once at startup and imported everywhere. Keeps sensitive config in one place.

  ---
  database.py

  Purpose: Connects to MongoDB and defines all collection references.

  Contains:
  - MongoDB client connection
  - All collection variables (users_collection, tasks_collection, categories_collection, etc.)

  Why separate: Any file can do from database import users_collection instead of setting up the connection themselves.

  ---
  models.py

  Purpose: Defines the shape of data going in/out of the API.

  Contains:
  - Pydantic models for request validation (UserCreate, Task, Review)
  - Pydantic models for responses (UserPublic, CategoryResponse)

  Why separate: Models are used across multiple routers. Centralizing them prevents circular imports.

  ---
  security.py

  Purpose: Authentication and authorization utilities.

  Contains:
  - hash_password() / verify_password() - Password hashing
  - create_access_token() - JWT token creation
  - get_current_user() - FastAPI dependency that validates tokens
  - serialize_document() - Converts MongoDB docs to JSON
