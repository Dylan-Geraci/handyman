# HANDYMAN PLATFORM - DATABASE SCHEMA & BACKEND STRUCTURE

## DATABASE SCHEMA

### Collections Overview
The platform uses MongoDB with 10 collections:
1. users - User accounts (clients/taskers)
2. tasks - Job postings
3. reviews - Task ratings
4. messages - Chat messages
5. notifications - User notifications
6. services - Admin service listings
7. portfolio_items - Admin portfolio
8. contact_requests - Contact form submissions
9. categories - Task categories
10. task_types - Specific task types within categories

## COLLECTION SCHEMAS

### 1. users
```
{
  _id: ObjectId,
  username: string,
  full_name: string,
  hashed_password: string,
  role: string ("client" | "tasker"),
  location: string (optional),
  skills: [string] (optional, taskers only),
  profile_image_url: string (optional),
  bio: string (optional),
  service_categories: [string] (optional, array of category_id references for taskers),
  created_at: datetime,

  // Recommendation system fields (taskers only):
  experience_level: int (1=beginner, 2=intermediate, 3=expert),
  completed_tasks_count: int (cached for performance),
  avg_rating: float (cached from reviews),
  preferred_locations: [string] (array of preferred work locations),

    lng: float

}
```

## DATABASE INDEXES

Performance indexes created for recommendation system:
1. tasks.posted_at (descending) - Fast sorting by recency
2. tasks.category_id + status (compound) - Efficient category filtering
3. tasks.location (text) - Location-based search
4. users.service_categories - Tasker skill matching
5. users.role + experience_level (compound) - Experience-based filtering

## BACKEND FILE STRUCTURE

### config.py
Purpose: Loads all settings and external service configurations.

Contains:
- Environment variables (SECRET_KEY, DATABASE_URL, GOOGLE_API_KEY)
- JWT settings (algorithm, expiration time)
- Password hasher setup
- Gemini AI model initialization

Why separate: Settings are loaded once at startup and imported everywhere. Keeps sensitive config in one place.

### database.py
Purpose: Connects to MongoDB and defines all collection references.

Contains:
- MongoDB client connection
- All collection variables (users_collection, tasks_collection, categories_collection, etc.)

Why separate: Any file can do `from database import users_collection` instead of setting up the connection themselves.

### models.py
Purpose: Defines the shape of data going in/out of the API.

Contains:
- Pydantic models for request validation (UserCreate, Task, Review)
- Pydantic models for responses (UserPublic, CategoryResponse)
- Recommendation models (RecommendationQuery, TaskRecommendation, MatchBreakdown)

Why separate: Models are used across multiple routers. Centralizing them prevents circular imports.

### security.py
Purpose: Authentication and authorization utilities.

Contains:
- hash_password() / verify_password() - Password hashing
- create_access_token() - JWT token creation
- get_current_user() - FastAPI dependency that validates tokens
- serialize_document() - Converts MongoDB docs to JSON

TLDR:

The AI recommendation system backend exceeds the PM's requirements:

 Scoring Algorithm

 A sophisticated 7-factor scoring system (0-100 scale) that considers:

 1. Type of Task (50 points total)
   - Category Match (20 pts) - Task category matches tasker's service categories
   - AI Semantic Match (30 pts) - Gemini AI analyzes task description vs tasker skills
 2. Distance to Task (25 points)
   - Geographic proximity with tiered scoring
   - 0-5 miles = 25pts, 5-10 miles = 20pts, 10-15 miles = 15pts, etc.
 3. Time Posted (15 points)
   - Recency-based scoring
   - Last hour = 15pts, 1-6 hours = 12pts, 1 day = 8pts, 3 days = 4pts
 4. Task Difficulty (10 points)
   - Matches task difficulty to tasker experience level
   - Perfect match = 10pts, adjacent level = 6pts
 5. Additional Smart Factors (15 points)
   - Historical Success (10 pts) - Past performance on similar tasks
   - Competition Level (5 pts) - Number of other qualified taskers nearby

 Total: All PM requirements met + bonus intelligence features

 API Endpoint

 - Route: GET /api/tasks/recommendations
 - Status: Fully functional and tested
 - Features:
   - Authenticated (taskers only)
   - Customizable via query parameters (limit, min_score, location_radius)
   - Returns scored tasks with detailed match breakdowns
   - Includes human-readable match reasons
